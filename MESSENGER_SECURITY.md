# Безопасность мессенджера Gomo6

## Обзор

Мессенджер Gomo6 использует end-to-end шифрование (E2EE) для защиты приватности сообщений. Сообщения шифруются на устройстве отправителя и расшифровываются только на устройстве получателя.

## Архитектура безопасности

### 1. End-to-End шифрование (E2EE)

**Используемая криптография:**
- Библиотека: `libsodium` (NaCl)
- Алгоритм: `crypto_box` (X25519 + XSalsa20 + Poly1305)
- Генерация ключей: `crypto_box_keypair()`

**Процесс:**
1. При первом входе в мессенджер генерируется пара ключей (публичный + приватный)
2. Публичный ключ сохраняется в БД (`chat_user_keys`)
3. Приватный ключ хранится ТОЛЬКО в localStorage браузера
4. При отправке сообщения:
   - Текст шифруется с использованием публичного ключа получателя и приватного ключа отправителя
   - Генерируется уникальный nonce для каждого сообщения
   - В БД сохраняется только зашифрованный текст (ciphertext) + nonce
5. При получении сообщения:
   - Клиент расшифровывает используя свой приватный ключ и публичный ключ отправителя

**Важно:** Сервер НИКОГДА не имеет доступа к приватным ключам и не может прочитать содержимое сообщений.

### 2. WebSocket безопасность

**Аутентификация:**
- WebSocket соединения требуют JWT токен
- Токен передаётся через query параметр `?token=...`
- Проверка токена происходит через middleware `SupabaseAuthMiddleware`

**CORS защита:**
- CheckOrigin проверяет источник WebSocket соединения
- Разрешённые origins настраиваются через переменную окружения `ALLOWED_ORIGINS`
- По умолчанию: `http://localhost:5173,http://localhost:8080,http://localhost:3000`

**Rate Limiting:**
- Ограничение: 60 сообщений в минуту на пользователя
- Применяется ко всем типам сообщений кроме ping
- При превышении лимита клиент получает ошибку

### 3. Защита базы данных

**Row Level Security (RLS):**
Все таблицы мессенджера защищены RLS политиками:

- `chat_user_keys`: Все могут читать публичные ключи, но изменять только свой
- `chat_conversations`: Пользователи видят только свои диалоги
- `chat_conversation_members`: Доступ только к своим членствам
- `chat_messages`: Чтение/запись только в диалогах где пользователь является участником
- `chat_receipts`: Доступ только к квитанциям своих сообщений

**SQL Injection защита:**
- Все запросы используют параметризованные запросы ($1, $2, etc.)
- Валидация UUID перед использованием

### 4. Дополнительные меры безопасности

**Хранение ключей:**
- Приватные ключи хранятся в localStorage с префиксом `gomo6-messenger-sodium-state`
- При смене пользователя ключи автоматически обновляются
- Ключи привязаны к userId для предотвращения путаницы

**Защита от replay атак:**
- Каждое сообщение имеет уникальный `client_message_id`
- Уникальный nonce для каждого шифрования
- Временные метки для всех сообщений

**Защита метаданных:**
Сервер видит только:
- Кто с кем общается (conversation_id)
- Когда отправлены сообщения (sent_at)
- Статусы доставки/прочтения

Сервер НЕ видит:
- Содержимое сообщений
- Приватные ключи пользователей

## Известные ограничения

### 1. Хранение приватного ключа в localStorage
**Риск:** Если злоумышленник получит доступ к localStorage (XSS атака), он может украсть приватный ключ.

**Митигация:**
- Используйте Content Security Policy (CSP)
- Регулярно обновляйте зависимости
- Проводите аудит кода на XSS уязвимости

**Альтернативы (для будущих версий):**
- Использование Web Crypto API с non-extractable ключами
- Хранение ключей в IndexedDB с дополнительным шифрованием
- Использование аппаратных токенов (WebAuthn)

### 2. Отсутствие Perfect Forward Secrecy (PFS)
**Риск:** Если приватный ключ скомпрометирован, все прошлые сообщения могут быть расшифрованы.

**Решение (для будущих версий):**
- Реализовать протокол Double Ratchet (как в Signal)
- Периодическая ротация ключей

### 3. Отсутствие верификации ключей
**Риск:** Man-in-the-middle атака при обмене публичными ключами.

**Решение (для будущих версий):**
- Добавить QR-коды для верификации ключей
- Safety numbers (как в Signal/WhatsApp)
- Key transparency logs

### 4. Метаданные не защищены
**Риск:** Сервер знает кто с кем общается и когда.

**Решение (для будущих версий):**
- Использование Sealed Sender (как в Signal)
- Onion routing для скрытия IP адресов

## Рекомендации по развёртыванию

### Production настройки

1. **HTTPS обязателен:**
```bash
# Используйте reverse proxy (nginx/caddy) с SSL
# Никогда не используйте ws:// в production, только wss://
```

2. **Настройте ALLOWED_ORIGINS:**
```bash
ALLOWED_ORIGINS=https://yourdomain.com,https://www.yourdomain.com
```

3. **Используйте сильные секреты:**
```bash
JWT_SECRET=$(openssl rand -base64 32)
FEDERATION_KEY=$(openssl rand -base64 32)
```

4. **Настройте CSP заголовки:**
```nginx
add_header Content-Security-Policy "default-src 'self'; script-src 'self' 'unsafe-inline' 'unsafe-eval'; style-src 'self' 'unsafe-inline';";
```

5. **Включите rate limiting на уровне nginx:**
```nginx
limit_req_zone $binary_remote_addr zone=websocket:10m rate=10r/s;
limit_req zone=websocket burst=20 nodelay;
```

6. **Мониторинг:**
- Логируйте все попытки подключения с неразрешённых origins
- Отслеживайте rate limit violations
- Мониторьте аномальную активность

## Аудит безопасности

### Что проверять:

1. **Шифрование:**
   - ✅ Используется libsodium (проверенная библиотека)
   - ✅ Уникальный nonce для каждого сообщения
   - ✅ Приватные ключи не покидают клиент

2. **Аутентификация:**
   - ✅ JWT токены для WebSocket
   - ✅ Middleware проверка на всех защищённых endpoints
   - ✅ Валидация UUID

3. **Авторизация:**
   - ✅ RLS политики на всех таблицах
   - ✅ Проверка владения перед операциями
   - ✅ Изоляция данных между пользователями

4. **Защита от атак:**
   - ✅ Rate limiting
   - ✅ CORS/Origin проверка
   - ✅ SQL injection защита (параметризованные запросы)
   - ⚠️ XSS защита (требует CSP на фронтенде)
   - ⚠️ CSRF защита (WebSocket не подвержен, но REST API требует проверки)

## Отчёт об уязвимостях

2. Свяжитесь с разработчиками напрямую
3. Дайте время на исправление перед публичным раскрытием
