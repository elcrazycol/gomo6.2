# Plan: Move file reads from Go backend to direct Garage via Caddy

## Problem

Every file request (images, avatars, attachments) flows: **Browser → Caddy → Go backend → Garage**. The backend streams files from Garage through itself, wasting CPU and memory on binary data it doesn't need to touch. Uploads need the backend (auth, validation), but reads don't.

## Goal

**Reads**: Browser → Caddy → Garage (direct)  
**Writes**: Browser → Caddy → Go backend → Garage (unchanged)

## Changes

### 1. Enable Garage `[website]` mode for public HTTP serving

**File**: `apps/backend-go/garage.toml`

Add a `[website]` section so Garage can serve public bucket content over HTTP without S3 signatures:

```toml
[website]
bind_addr = "0.0.0.0:3902"
```

### 2. Expose Garage website port in Docker

**File**: `docker-compose.yml`

Add port `3902:3902` to the `garage` service so Caddy (which runs in the same Docker network) can reach it. No need to expose it to host.

### 3. Make buckets publicly readable

**File**: `docker-compose.yml` (garage-init script, lines 113-128)

After bucket creation, grant public visibility:

```bash
$$G bucket modify "$$bucket" --visibility public
```

This allows anonymous HTTP reads via Garage's website port. The backend still writes via S3 API with credentials.

### 4. Route reads to Garage in Caddy

**File**: `Caddyfile`

Replace the current catch-all `/storage/*` → backend with method-based routing:

```caddy
# GET/HEAD requests → Garage directly (no backend)
@storageGet {
    path /storage/v1/object/*
    method GET HEAD
}
handle @storageGet {
    handle_path /storage/v1/object/* {
        header {
            Access-Control-Allow-Origin "*"
            Access-Control-Allow-Methods "GET, HEAD, OPTIONS"
            Access-Control-Allow-Headers "Content-Type, Range"
            Access-Control-Expose-Headers "Content-Length, Content-Range, Content-Type, Accept-Ranges"
            Cache-Control "public, max-age=3600"
            Accept-Ranges "bytes"
        }
        reverse_proxy garage:3902
    }
}

# Upload + DELETE → backend (auth required)
handle /storage/v1/upload {
    reverse_proxy backend:8080
}
@storageWrite {
    path /storage/v1/object/*
    method DELETE
}
handle @storageWrite {
    reverse_proxy backend:8080
}
```

`handle_path` strips `/storage/v1/object/` prefix, so a request to `/storage/v1/object/content/img.jpg` becomes `/content/img.jpg` — exactly what Garage's website mode expects.

### 5. Handle avatar placeholder fallback via Caddy

**Current behavior**: When `GET /storage/v1/object/post-images/*avatar*` returns 404, the backend returns an SVG placeholder (`storage.AvatarPlaceholderSVG`).

**New behavior**: Caddy's `handle_errors` block catches 404s from Garage. For avatar-related paths, serve the SVG placeholder. For other paths, return standard 404.

```caddy
handle_errors {
    @avatar404 {
        {err.status_code} == 404
        path /storage/v1/object/post-images/*avatar*
    }
    handle @avatar404 {
        root * /srv
        file /default-avatar.svg
    }
    handle {
        respond "{err.status_code} {err.message}" {err.status_code}
    }
}
```

The SVG file (`default-avatar.svg`) gets copied into the Caddy container's `/srv` directory during build or via a volume mount. The SVG content comes from `apps/backend-go/internal/storage/avatar_placeholder.go` (`AvatarPlaceholderSVG` constant).

### 6. Remove `ServeObject` from Go backend (optional cleanup)

**File**: `apps/backend-go/internal/storage/handlers/upload.go`  
**File**: `apps/backend-go/internal/api/routes/routes.go`

The `ServeObject` handler and the `GET /storage/v1/object/:bucket/*key` route become unused after Caddy serves reads directly. They can be removed in a follow-up to keep this change minimal, or kept as a fallback during migration.

### 7. Update Caddyfile comment

**File**: `Caddyfile` (line 21-22)

Replace the comment "Garage is never exposed directly to browsers" with accurate documentation of the new architecture.

## What stays unchanged

- Frontend `storage.ts` — URL pattern `/storage/v1/object/{bucket}/{key}` stays identical
- Upload flow — POST `/storage/v1/upload` → backend → Garage (unchanged)
- Delete flow — DELETE `/storage/v1/object/:bucket/*key` → backend → Garage (unchanged)
- All 30+ components using `storageUrl()` — no changes needed
