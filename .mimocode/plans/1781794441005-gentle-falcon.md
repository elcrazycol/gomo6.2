# Fix: Storage 404 — Garage direct-read routing broken

## Root Cause

Caddy strips `/storage/v1/object/` via `handle_path` and proxies the remaining path to `garage:3902` (Garage's `s3_web` website port). But Garage's website mode uses **virtual-hosted style** — it extracts the bucket name from the `Host` header using `root_domain = ".storage.garage.localhost"`.

Caddy sends `Host: garage` (Docker service name). Garage can't match any bucket → returns 404 HTML (51 bytes).

**Flow:**
```
Browser → GET /storage/v1/object/post-images/user/avatar.jpg
  → Caddy strips prefix → path = /post-images/user/avatar.jpg, Host = garage
  → Garage:3902 sees Host "garage" ≠ "<bucket>.storage.garage.localhost" → 404
```

## Fix: Caddy `path_regexp` + `header_up Host` rewrite

Use `path_regexp` to capture bucket name and key, then rewrite Host and path before proxying to Garage.

### File: `Caddyfile` (lines 100-117)

**Current (broken):**
```caddy
@storageGet {
    path /storage/v1/object/*
    method GET HEAD
}
handle @storageGet {
    handle_path /storage/v1/object/* {
        header {
            Access-Control-Allow-Origin "*"
            Access-Control-Allow-Methods "GET, HEAD, OPTIONS"
            Access-Control-Allow-Headers "Content-Type, Range"
            Access-Control-Expose-Headers "Content-Length, Content-Range, Content-Type, Accept-Ranges"
            Cache-Control "public, max-age=3600"
            Accept-Ranges "bytes"
        }
        reverse_proxy garage:3902
    }
}
```

**New (fixed):**
```caddy
@storageGet {
    method GET HEAD
    path_regexp storage_read ^/storage/v1/object/([^/]+)/(.+)$
}
handle @storageGet {
    route {
        header {
            Access-Control-Allow-Origin "*"
            Access-Control-Allow-Methods "GET, HEAD, OPTIONS"
            Access-Control-Allow-Headers "Content-Type, Range"
            Access-Control-Expose-Headers "Content-Length, Content-Range, Content-Type, Accept-Ranges"
            Cache-Control "public, max-age=3600"
            Accept-Ranges "bytes"
        }
        rewrite * /{re.storage_read.2}
        reverse_proxy garage:3902 {
            header_up Host "{re.storage_read.1}.storage.garage.localhost"
        }
    }
}
```

**What this does:**
1. `path_regexp storage_read ^/storage/v1/object/([^/]+)/(.+)$` — captures:
   - Group 1 (`{re.storage_read.1}`) = bucket name (e.g. `post-images`)
   - Group 2 (`{re.storage_read.2}`) = object key (e.g. `user/avatar.jpg`)
2. `rewrite * /{re.storage_read.2}` — path becomes `/{key}` (with leading slash, as Garage expects)
3. `header_up Host "{re.storage_read.1}.storage.garage.localhost"` — rewrites Host to virtual-hosted format
4. `reverse_proxy garage:3902` — forwards to Garage website port

**Resulting flow:**
```
Browser → GET /storage/v1/object/post-images/user/avatar.jpg
  → path_regexp captures: bucket=post-images, key=user/avatar.jpg
  → rewrite: path = /user/avatar.jpg
  → header_up: Host = post-images.storage.garage.localhost
  → Garage:3902 resolves bucket from Host → serves file ✓
```

## No other changes needed

- **Backend**: Storage routes unchanged. `ServeObject` still handles writes/fallbacks internally.
- **Frontend**: `storageUrl()` already constructs correct `/storage/v1/object/{bucket}/{key}` URLs.
- **Uploads**: Still Caddy → Backend → S3 API (port 3900). Unchanged.
- **Deletes**: Still Caddy → Backend → S3 API. Unchanged.
- **Garage config**: `root_domain = ".storage.garage.localhost"` already matches the rewritten Host.
- **Buckets**: Already set to `website` access via `garage bucket website --allow` in docker-compose init.

## Edge Cases

- **Key with slashes**: `/storage/v1/object/post-images/a/b/c.jpg` → key=`a/b/c.jpg` → path=`/a/b/c.jpg` ✓
- **No key** (just bucket): `/storage/v1/object/post-images` → regex doesn't match (`.+` requires 1+ chars) → falls through to catch-all handler → backend or 404. Correct behavior.
- **Leading slash in rewritten path**: `/{re.storage_read.2}` always has `/` prefix. Garage expects `/key`. ✓
- **CORS**: Headers set inside the `route` block, applied to all responses including 404s. ✓
