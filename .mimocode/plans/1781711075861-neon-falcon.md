# Plan: Gifts Feature (Подарки)

## Overview
Users can buy gifts from a catalog for garma and send them to other users. Gifts appear in the recipient's profile in a new "Подарки" tab. Admin manages the catalog via Dev Dashboard.

---

## 1. Database Migration (`migrations/047_gifts.sql`)

### `gift_catalog` — admin-managed gift definitions
```sql
CREATE TABLE gift_catalog (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    name VARCHAR(255) NOT NULL,
    description TEXT,
    image_url TEXT NOT NULL,
    price INTEGER NOT NULL CHECK (price > 0),
    category VARCHAR(100) DEFAULT 'general',
    is_active BOOLEAN DEFAULT TRUE,
    is_limited BOOLEAN DEFAULT FALSE,
    max_quantity INTEGER,
    sold_count INTEGER DEFAULT 0,
    sort_order INTEGER DEFAULT 0,
    created_at TIMESTAMPTZ DEFAULT NOW(),
    updated_at TIMESTAMPTZ DEFAULT NOW()
);
```

### `user_gifts` — received gifts
```sql
CREATE TABLE user_gifts (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    gift_id UUID NOT NULL REFERENCES gift_catalog(id),
    sender_id UUID REFERENCES users(id) ON DELETE SET NULL,
    recipient_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    message TEXT,
    is_anonymous BOOLEAN DEFAULT FALSE,
    created_at TIMESTAMPTZ DEFAULT NOW()
);

CREATE INDEX idx_user_gifts_recipient ON user_gifts(recipient_id);
CREATE INDEX idx_user_gifts_sender ON user_gifts(sender_id);
```

Key decisions:
- `sender_id ON DELETE SET NULL` — if sender deletes account, gift stays but sender becomes "Удалённый аккаунт"
- `recipient_id ON DELETE CASCADE` — if recipient deletes account, their gifts are cleaned up
- No unique constraint on (gift_id, recipient_id) — unlimited standard gifts, multiple of same gift allowed

## 2. Backend Changes

### 2a. Models (`internal/models/models.go`)
Add:
- `GiftCatalog` struct (matches gift_catalog table)
- `UserGift` struct (matches user_gifts table, with joined gift catalog data)
- `SendGiftRequest` struct (`gift_id`, `recipient_id`, `message?`, `is_anonymous`)

### 2b. Gift Handler (`internal/api/handlers/gifts.go`) — NEW FILE
- `GiftsHandler` struct with `db`, `redis`, `achievementChecker` fields
- `SetRedis(redis)` and `SetWebSocketHub(hub)` setters

**`SendGift(c *gin.Context)`** — POST protected endpoint:
1. Validate: gift exists & is_active, recipient exists, sender != recipient
2. Atomic transaction (anti-race-condition pattern):
```sql
-- Step 1: Atomically reserve limited gift (prevents race condition)
UPDATE gift_catalog
SET sold_count = sold_count + 1
WHERE id = $1 AND is_active = true
  AND (is_limited = false OR sold_count < max_quantity);
-- If rows_affected == 0 → ROLLBACK, return "gift sold out" error

-- Step 2: Deduct garma (atomic UPDATE with WHERE check)
UPDATE users SET garma = garma - $1
WHERE id = $2 AND garma >= $1;
-- If rows_affected == 0 → ROLLBACK, return "insufficient garma" error

-- Step 3: Insert user_gift
INSERT INTO user_gifts (gift_id, sender_id, recipient_id, message, is_anonymous)
VALUES ($1, $2, $3, $4, $5);
```
3. Send notification to recipient (type `gift_received`) + WebSocket push
4. Invalidate cache for recipient profile + user_gifts

**`GetUserGifts(c *gin.Context)`** — GET public endpoint:
- Query param: `recipient_id` (required)
- Pagination: `limit` (default 50, max 100), `offset` (default 0)
- Join user_gifts + gift_catalog, ORDER BY created_at DESC
- If is_anonymous: hide sender_id/sender_username in response

**`GetGiftCatalog(c *gin.Context)`** — GET public endpoint:
- Return active gifts, ORDER BY sort_order ASC, created_at DESC
- Pagination support

### 2c. Routes (`internal/api/routes/routes.go`)
Add under `rest` group (public):
```
rest.GET("/gift_catalog", giftsHandler.GetGiftCatalog)
rest.GET("/user_gifts", giftsHandler.GetUserGifts)
```
Add under `protected` group:
```
protected.POST("/gifts/send", giftsHandler.SendGift)
```

### 2d. Cache Invalidation
- Add `"gift_catalog"` and `"user_gifts"` cases to `invalidateCacheForTableResult` in `universal_crud.go`
- **Do NOT add universal CRUD routes for `gift_catalog`** — admin management goes through Dev Dashboard with admin auth middleware, not through public universal CRUD. This prevents garma price manipulation via public API.

### 2e. Wire up in routes.go
```go
giftsHandler := handlers.NewGiftsHandler(db)
giftsHandler.SetRedis(redis)
giftsHandler.SetWebSocketHub(wsHub)
```

### 2f. Dev Dashboard Admin API (`internal/api/handlers/gift_admin.go`) — NEW FILE
Protected by admin-only middleware. CRUD for gift_catalog:
- `GET /api/v1/admin/gifts` — list all gifts (including inactive)
- `POST /api/v1/admin/gifts` — create gift
- `PUT /api/v1/admin/gifts/:id` — update gift
- `DELETE /api/v1/admin/gifts/:id` — soft-delete (set is_active=false) or hard delete

## 3. Frontend Changes

### 3b. New Components (`apps/web/src/components/`)

**`GiftCard.tsx`** — displays a single gift:
- Image, name, price (garma), description
- Two variants: `catalog` (with buy button) and `received` (with sender info + date)

**`GiftSendDialog.tsx`** — Dialog for sending a gift:
- Shows gift image + name + price
- Optional message textarea
- Anonymous toggle switch
- Confirm button → **disable + show loader immediately on click** (prevent double-submit)
- Success/error toast, close dialog on success

**`GiftsTab.tsx`** — renders the gifts list on profile:
- Grid of received gifts (GiftCard variant=received)
- Each shows: gift image, "от @username" (or "Аноним"), message, date
- Empty state: "Подарков пока нет"
- Load-more button for pagination

- Add `gifts` to `activeTab` union type: `'wall' | 'achievements' | 'threads' | 'gifts'`
- Add state: `userGifts`, `giftsLoading`
- Add "Подарки" tab button in the tab bar
- Add `loadUserGifts` function (fetch `/api/v1/user_gifts?recipient_id=eq.${userId}`)
- Render `<GiftsTab>` when activeTab === 'gifts'
- Add "Подарить" button on other users' profiles (next to "Написать"), opens GiftSendDialog
- Add `giftCatalog` state + `showGiftDialog` state for the send dialog

### 3d. Dev Dashboard (`apps/dev-dashboard/`)
- New page: `/gifts` — Gift catalog management

- Create/Edit form with image upload

## 5. Security & Edge Cases

- **Race condition on limited gifts**: Atomic `UPDATE ... WHERE sold_count < max_quantity` inside TX
- **Admin API protection**: gift_catalog not exposed via universal CRUD; admin endpoints use separate auth
- **Account deletion**: `sender_id ON DELETE SET NULL` preserves gift history
- **Double-submit**: Frontend disables button + shows loader on click
- **Pagination**: All list endpoints support limit/offset (default 50, max 100)
- **Sort order fallback**: Catalog sorts by `sort_order ASC, created_at DESC`
