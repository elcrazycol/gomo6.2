# Plan: Sistema druzey (Friends System)

## Обзор

Двусторонняя система друзей: заявка -> принятие/отклонение -> друзья. Интеграция с профилем и мессенджером.

## 1. База данных (migration 055)

**Файл:** `apps/backend-go/migrations/055_friends.sql`

```sql
-- Таблица заявок в друзья (исторически: хранит все заявки, включая принятые)
CREATE TABLE friend_requests (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    sender_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    receiver_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    status VARCHAR(20) NOT NULL DEFAULT 'pending', -- pending, accepted, rejected
    created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
    updated_at TIMESTAMP WITH TIME ZONE DEFAULT NOW()
);

-- Уникальность только для pending-заявок (чтобы не спамить)
CREATE UNIQUE INDEX idx_unique_pending_request 
    ON friend_requests(sender_id, receiver_id) 
    WHERE status = 'pending';

-- Таблица дружеских связей (создается при принятии заявки, удаляется при разрыве)
CREATE TABLE friendships (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user1_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    user2_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
    UNIQUE(user1_id, user2_id),
    CHECK (user1_id < user2_id) -- для уникальности пары
);

-- Индексы
CREATE INDEX idx_friend_requests_receiver ON friend_requests(receiver_id, status);
CREATE INDEX idx_friend_requests_sender ON friend_requests(sender_id, status);
CREATE INDEX idx_friendships_user1 ON friendships(user1_id);
CREATE INDEX idx_friendships_user2 ON friendships(user2_id);
```

**Ключевые решения:**
- `friend_requests` НЕ удаляется при принятии — статус меняется на `accepted`. Это сохраняет историю и решает проблему повторной отправки.
- При удалении из друзей: удаляем `friendship` + переводим `friend_requests` в статус `rejected` (или оставляем `accepted` как историю).
- `UNIQUE` индекс только для `pending` — не мешает логике.

### Логика AcceptRequest (транзакция!):

```go
func (h *FriendsHandler) AcceptRequest(c *gin.Context) {
    tx, _ := h.db.Begin()
    defer tx.Rollback()
    
    // 1. Пометить заявку как accepted
    tx.Exec("UPDATE friend_requests SET status='accepted' WHERE id=$1 AND receiver_id=$2", requestID, userID)
    
    // 2. Создать friendships (user1 < user2)
    tx.Exec("INSERT INTO friendships (user1_id, user2_id) VALUES ($1, $2)", min(sender, receiver), max(sender, receiver))
    
    // 3. Уведомление отправителю
    CreateNotification(..., "friend_accepted", "@username принял вашу заявку в друзья")
    
    tx.Commit()
}
```

### Логика RemoveFriend:

```go
func (h *FriendsHandler) RemoveFriend(c *gin.Context) {
    targetUserID := c.Param("userId")
    
    // 1. Удалить friendship
    h.db.Exec("DELETE FROM friendships WHERE (user1_id=$1 AND user2_id=$2) OR (user1_id=$2 AND user2_id=$1)", currentUserID, targetUserID)
    
    // 2. (Опционально) Очистить старые заявки — чтобы можно было отправить заново
    h.db.Exec("UPDATE friend_requests SET status='rejected' WHERE ((sender_id=$1 AND receiver_id=$2) OR (sender_id=$2 AND receiver_id=$1)) AND status='accepted'", currentUserID, targetUserID)
}
```

### Валидация:
- Нельзя отправить заявку самому себе
- Нельзя отправить, если уже есть pending заявка (UNIQUE индекс)
- Нельзя отправить, если уже друзья
- При отправке реверсивной заявки — автоматически принимаем (оба хотят быть друзьями)

## 5. Frontend - Zustand store

**Новый файл:** `apps/web/src/stores/friendsStore.ts`

```typescript
type FriendStatus = 'none' | 'pending_sent' | 'pending_received' | 'friends';

interface FriendsStore {
    friends: Friend[];
    incomingRequests: FriendRequest[];
    outgoingRequests: FriendRequest[];
    friendStatusMap: Record<string, FriendStatus>; // userId -> status
    
    fetchFriends: () => Promise<void>;
    fetchRequests: () => Promise<void>;
    sendRequest: (userId: string) => Promise<void>; // optimistic
    acceptRequest: (requestId: string) => Promise<void>; // optimistic
    rejectRequest: (requestId: string) => Promise<void>; // optimistic
    removeFriend: (userId: string) => Promise<void>; // optimistic
    checkStatus: (userId: string) => Promise<FriendStatus>;
    setStatus: (userId: string, status: FriendStatus) => void;
}
```

**Ключевые моменты:**
- **Optimistic updates** для мгновенного UX: при `sendRequest` сразу меняем статус на `pending_sent`, при ошибке — откат
- `friendStatusMap` — кэш, чтобы не перезапрашивать статус на каждой странице
- Инвалидация кэша **обеих** пользователей при мутациях

При операциях с друзьями инвалидировать кэш **обеих** пользователей:

```go
func invalidateFriendCaches(redis *redis.Client, user1ID, user2ID string) {
    // Инвалидируем data cache для обоих
    middleware.InvalidateCacheForTable(redis, "friend_requests")
    middleware.InvalidateCacheForTable(redis, "friendships")
    // Или более точно:
    patterns := []string{
        "data:/api/v1/friends*",
    }
    // Удаляем ключи для обоих пользователей
}
```

**Важно:** `universal_crud.go` обычно инвалидирует по ID текущего пользователя. Для friends нужно вызывать инвалидацию и для `receiver_id` тоже. Это делается в handler, а не в universal_crud.
