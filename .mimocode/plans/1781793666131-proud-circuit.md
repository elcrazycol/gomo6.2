# Fix: "Failed to record transaction" when sending gifts

## Problem

Gift sending (`POST /api/v1/gifts/send`) fails with HTTP 500 and `"Failed to record transaction"`.

The error occurs at `gifts.go:131-136` when inserting into `drops_transactions`:

```sql
INSERT INTO drops_transactions (user_id, type, amount, balance_after, reference_id, reference_type, description)
VALUES ($1, 'gift_send', -$2, $3, $4::uuid, 'gift', $5)
```

The `drops_transactions` table was created in migration `048_drops.sql` with `reference_id UUID` and `reference_type VARCHAR(50)` columns. However, if the migration was already applied before those columns were added to the file (or applied from an earlier version), `CREATE TABLE IF NOT EXISTS` skips the table and the columns are missing. This is a classic `CREATE TABLE IF NOT EXISTS` pitfall — the columns exist in the migration file now but the live table doesn't have them.

## Root cause

The `drops_transactions` table in production likely lacks `reference_id` and `reference_type` columns. The `CREATE TABLE IF NOT EXISTS` in migration 048 was already applied before the gift system needed these columns, so they were never added.

## Fix

### 1. Add a new migration to ensure columns exist

Create `migrations/050_drops_transactions_add_reference_columns.sql`:

```sql
ALTER TABLE drops_transactions ADD COLUMN IF NOT EXISTS reference_id UUID;
ALTER TABLE drops_transactions ADD COLUMN IF NOT EXISTS reference_type VARCHAR(50);
```

### 2. Add error logging in the gift handler

In `gifts.go:135`, log the actual database error before returning the generic message, so future issues are diagnosable:

```go
log.Printf("[Gifts] record transaction error: %v", err)
```

### Files to modify

- `apps/backend-go/migrations/050_drops_transactions_add_reference_columns.sql` (new)
- `apps/backend-go/internal/api/handlers/gifts.go` (add log at line ~135)

### Verification

1. `cd apps/backend-go && go build ./...` — confirm compilation
2. `docker compose up -d` locally (or wait for CI deploy) — migration 050 applies
3. Try sending a gift again — should succeed
4. Check backend logs for actual error messages if it still fails
