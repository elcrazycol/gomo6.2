# Fix: Gift notification leaks to sender

## Problem

Sometimes the sender of a gift receives a notification about gifting themselves. The notification should only go to the recipient.

## Root Cause Analysis

The backend logic is correct:
- `gifts.go:162` — `sendGiftNotification(recipientID, senderID, ...)` correctly passes recipient first
- `gifts.go:200-204` — DB insert uses `recipientID` as `user_id`
- `gifts.go:215-224` — WS event has `user_id: recipientID`
- `hub.go:300-304` — Routes to `notifications_{recipientID}` room
- `client.go:107` — Clients auto-subscribe to `notifications_{ownUserID}`

**The gap**: The frontend `handleWsNotification` (`notificationStore.ts:71-73`) blindly accepts ANY `new_notification` WS message and adds it to state **without verifying `notif.user_id` matches the current user**:

```typescript
// notificationStore.ts:71-73
const handleWsNotification = (message: WebSocketMessage) => {
  const notif = message.data as Notification & { achievement?: AchievementData };
  if (!notif || !notif.id) return;  // ← NO user_id check
```

This means if any edge case in WS delivery (reconnection, room leak, Redis fan-out) delivers a notification to the wrong client, the frontend will display it. The intermittent nature ("иногда") confirms a timing/delivery edge case.

Additionally, `InvalidateCacheForNotification` (`data_cache.go:229`) uses an overly broad wildcard `"data:/api/v1/notifications*"` that invalidates ALL users' notification caches — not the root cause, but wasteful.

## Changes

### 1. Frontend defense-in-depth: validate `user_id` on incoming WS notification

**File**: `apps/web/src/stores/notificationStore.ts:71-73`

Add a check that `notif.user_id` matches the current user before adding to state. The `userId` is available from the `init` call parameter (already captured in `initializedUserId`).

```typescript
const handleWsNotification = (message: WebSocketMessage) => {
  const notif = message.data as Notification & { achievement?: AchievementData };
  if (!notif || !notif.id) return;

  // Defense-in-depth: ignore notifications for other users
  if (notif.user_id && notif.user_id !== initializedUserId) return;
  ...
```

### 2. Backend: narrow cache invalidation pattern

**File**: `apps/backend-go/internal/middleware/data_cache.go:227-229`

Remove the overly broad `"data:/api/v1/notifications*"` pattern from `InvalidateCacheForNotification`. Only invalidate the specific user's cache:

```go
func InvalidateCacheForNotification(redisClient *redis.Client, userID string) {
    patterns := []string{
        fmt.Sprintf("data:/api/v1/notifications*user_id=eq.%s*", userID),
    }
    ...
}
```
