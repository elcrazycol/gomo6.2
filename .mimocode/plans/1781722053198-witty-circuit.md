# Plan: Refactor Notification System

## Problem

Three completely independent notification consumers (NotificationBell, Notify page, AchievementToastListener) each manage their own state, polling, and WS listeners. Result: notifications appear/disappear randomly, badge count desyncs, the full notifications page is stale, and nothing is unified.

## Root Causes

1. **No shared state** ÔÇö Each component has its own `useState` for notifications/unreadCount
2. **Notify page has zero real-time** ÔÇö No WS, no polling, static after initial load
3. **Count desync** ÔÇö Optimistic decrements in bell get overridden on WS reconnect when re-fetching `/unread-count`
4. **Inconsistent polling** ÔÇö Bell: 60s, AchievementToast: 30s, both only when WS disconnected
5. **Bell links wrong** ÔÇö Links to `/notify?thread=UUID` instead of actual board/thread URLs
6. **AchievementToastListener uses raw `fetch()`** ÔÇö Bypasses apiClient

## Solution: Unified Zustand Store

Create `apps/web/src/stores/notificationStore.ts` ÔÇö single source of truth. All consumers read from it. Store manages WS subscription, polling, fetch, mark-as-read.

## Step 1: Create `notificationStore.ts`

**New file:** `apps/web/src/stores/notificationStore.ts`

Zustand store with:
- `notifications: Notification[]` ÔÇö latest notifications (fetched from server)
- `unreadCount: number` ÔÇö from `/unread-count` endpoint
- `isLoading: boolean` ÔÇö initial load state
- `initialized: boolean` ÔÇö whether WS subscription is set up

Actions:
- `init(userId)` ÔÇö subscribe to WS room, start polling fallback (15s when disconnected), fetch initial data. Only runs once.
- `fetchNotifications(limit?)` ÔÇö GET `/api/v1/notifications?limit=N`, updates state
- `fetchUnreadCount()` ÔÇö GET `/api/v1/notifications/unread-count`, updates `unreadCount`
- `handleNewNotification(notif)` ÔÇö WS handler: prepend to notifications, increment unreadCount if unread
- `markAsRead(id)` ÔÇö optimistic local update + PUT `/api/v1/notifications/:id/read`
- `markAllAsRead()` ÔÇö optimistic local update + PUT `/api/v1/notifications/read-all`
- `cleanup()` ÔÇö unsub WS, clear polling interval

Key behaviors:
- On WS `connected` event: re-subscribe to notification room, immediately fetch unread count
- Polling: `setInterval(15000)` ÔÇö only when `wsService.connected === false`
- WS reconnect triggers fresh `fetchUnreadCount()` to fix any drift
- Dedup on `handleNewNotification` by notification id

## Step 2: Refactor NotificationBell.tsx

**Modify:** `apps/web/src/components/NotificationBell.tsx`

Remove:
- All local state (`notifications`, `unreadCount`)
- All `useEffect` for WS subscription and polling
- `loadNotifications`, `loadUnreadCount` callbacks
- WS reconnect handler

Replace with:
- `const { notifications, unreadCount, markAsRead } = useNotificationStore()`
- `useEffect` that calls `init(userId)` on mount
- Render `notifications.slice(0, 5)` for the dropdown
- `markAsRead` calls store method
- Bell links: keep `/notify?thread=UUID` for now (slug resolution is complex and not blocking)

The component becomes ~100 lines ÔÇö just UI + store reads.

## Step 3: Refactor Notify.tsx

**Modify:** `apps/web/src/pages/Notify.tsx`

Remove:
- Local `notifications` state
- `loadNotifications` callback (the fetching logic)

Replace with:
- `const { notifications, markAsRead, markAllAsRead, fetchNotifications } = useNotificationStore()`
- `useEffect` that calls `fetchNotifications(PAGE_SIZE)` on mount
- Infinite scroll calls `fetchNotifications` with offset ÔÇö but since the store holds a flat list, implement a `fetchMoreNotifications(offset)` action in the store that appends results
- Mark-all-as-read calls `markAllAsRead()` from store ÔÇö Bell immediately sees 0 unread
- The page is now always in sync with the bell because both read from the same store

Actually ÔÇö better approach for Notify: keep local pagination state in the component (offset, hasMore, loadingMore) since it's page-specific. But use the store for:
- The initial batch of notifications
- Mark-as-read / mark-all-as-read (propagates to bell)
- Unread count

Simpler: Notify keeps its own paginated list for infinite scroll, but when a notification is marked as read (individual or all), it calls the store action which syncs everywhere.

## Step 4: Refactor AchievementToastListener.tsx

**Modify:** `apps/web/src/components/AchievementToastListener.tsx`

Remove:
- Its own WS subscription for `new_notification`
- Its own polling logic (setInterval with raw fetch)
- `shownNotificationIds` Set

Replace with:
- Subscribe to store's `notifications` changes
- When a new `achievement_unlock` notification appears in the store, show toast
- Use `shownNotificationIds` still (or add a `lastSeenId` in store) to prevent re-showing
- `markNotificationRead` still calls apiClient directly (it's fire-and-forget, and the store will also handle it)

Actually ÔÇö AchievementToastListener is specialized enough that it can remain mostly independent. The key fix: replace its raw `fetch()` polling with consuming from the store's notification stream. When the store receives a new achievement_unlock notification via WS, it fires a callback that the listener subscribes to.

Simplest approach: AchievementToastListener keeps its WS listener (for the achievement data which includes extra fields not in the standard Notification type), but removes its polling fallback. The store handles delivery; AchievementToastListener just watches for achievement_unlock events.
