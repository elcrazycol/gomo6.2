# Fix: ThreadCard image_urls not wrapped with storageUrl()

## Problem
After S3 storage migration (Garage), images load fine in profiles, gomosubs, board pages, thread pages — but are **broken on the main page** (recommendations feed, subscriptions feed). The `<img>` renders a raw storage key (e.g. `f0123c5d-4ecf-4d4a-8079-0f56a7939b31/1781805882931_mr666ub0998.JPG`) as the `src` attribute, producing a broken image.

## Root Cause
**`apps/web/src/components/ThreadCard.tsx:352`** — The `image_urls` array is rendered with raw `src={url}` instead of `src={storageUrl("content", url)}`. Every other component in the codebase wraps image keys through `storageUrl()`, but ThreadCard does not.

- `storageUrl` is already imported (line 7) and used for avatars in the same file (line 239).
- `image_urls` from the backend are storage keys (not full URLs), so they need wrapping.

## Fix

### File: `apps/web/src/components/ThreadCard.tsx`
**Line 352**: Change `src={url}` to `src={storageUrl("content", url) || url}`

```diff
-                    src={url}
+                    src={storageUrl("content", url) || url}
```

This follows the exact same pattern used in every other component:
- `ThreadAttachments.tsx:34` — `storageUrl("content", att.url) || att.url`
- `WallAttachments.tsx:18` — `storageUrl("content", attachment.url) || attachment.url`
- `Board.tsx:1425` — `storageUrl("content", thread.image_url) || thread.image_url`
- `ModerationPosts.tsx:386` — `storageUrl("content", content.thread.image_url) || content.thread.image_url`

No other files need changes — this is the only place `image_urls` is rendered without `storageUrl()`.
