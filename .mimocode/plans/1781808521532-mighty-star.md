# Fix Notification System — Bulletproof Notifications

## Problem Summary
Notifications sometimes don't arrive, arrive late, show stale data, or show phantom "new" indicators. Root causes: missing cache invalidation for gift/achievement notifications, WS lifecycle leaks, no dedup between WS push and REST fetch, disconnect nukes all handlers.

## Changes (7 files, ordered by implementation)

### 1. Backend: Gift notification — add cache invalidation + DB-returned created_at
**File:** `apps/backend-go/internal/api/handlers/gifts.go`
- Add `"github.com/gomo6/backend/internal/middleware"` import
- After the INSERT, add: `middleware.InvalidateCacheForNotification(h.redis, recipientID)`
- Change the INSERT query to `RETURNING id, created_at` and scan `created_at` into a variable
- Use the DB-returned timestamp in the WS payload: `"created_at": createdAt.Format(time.RFC3339Nano)` — NOT `time.Now()` (avoids clock drift between DB and Go runtime)

### 2. Backend: Achievement notification — fix types + add cache invalidation + DB-returned created_at
**File:** `apps/backend-go/internal/api/handlers/achievement_checker.go`
- Check for circular import: `achievement_checker.go` is in `handlers` package, `websocket.Hub` is in `websocket` package → no cycle, safe to use concrete types
- Change `redis interface{}` → `redis *redis.Client` (line 16)
- Change `wsHub interface{}` → `wsHub *websocket.Hub` (line 17)
- Change `SetRedis(redis interface{})` → `SetRedis(redis *redis.Client)` (line 26)
- Change `SetWebSocketHub(hub interface{})` → `SetWebSocketHub(hub *websocket.Hub)` (line 29)
- Add imports: `"github.com/gomo6/backend/internal/middleware"`, `"github.com/redis/go-redis/v9"`
- After the INSERT, add: `middleware.InvalidateCacheForNotification(ac.redis, userID)`
- Remove the type assertion `if hub, ok := ac.wsHub.(*websocket.Hub); ok {` — use `ac.wsHub` directly
- Change INSERT to `RETURNING id, created_at`, scan `createdAt`, use it in WS payload — NOT `time.Now()`

### 3. Frontend: notificationStore — lifecycle, dedup, rollback, throttled reconnect re-fetch
**File:** `apps/web/src/stores/notificationStore.ts`
- **init() cleanup first:** Before `initializedUserId = userId`, call old unsubscribe/clearInterval if `initializedUserId !== null` (prevents leaked handlers on re-login)
- **fetchInitial dedup + sort:** After fetching REST data, merge with existing state:
  ```
  wsOnly = currentNotifications.filter(item => !restData.some(r => r.id === item.id))
  merged = [...wsOnly, ...restData].sort(by created_at DESC)
  ```
  Prevents overwriting real-time pushes and ensures correct ordering.
- **WS reconnect re-fetch (throttled):** In the `connected` handler (line ~102), add `fetchInitial()` but wrapped in a throttle (min 2s between calls) to prevent storm on flaky connections that reconnect rapidly
- **markAsRead rollback:** Save previous state before optimistic update, restore on API failure
- **markAllAsRead rollback:** Same pattern

### 4. Frontend: Cleanup on logout
**File:** `apps/web/src/integrations/api/auth.ts`

- Add import for `useNotificationStore`

### 5. Frontend: WS disconnect — don't nuke all handlers
**File:** `apps/web/src/services/websocket.ts`
- Remove `this.messageHandlers.clear()` from `disconnect()` (line 412). Handlers are managed by components via their own unsubscribe functions.

### 6. Frontend: NotificationBell — remove redundant slice, add cleanup
**File:** `apps/web/src/components/NotificationBell.tsx`
- Remove redundant `.slice(0, 5)` on line 87 (already sliced on line 16)
- Add cleanup on unmount in the `useEffect` (return `cleanup()`)

### 7. Frontend: Notify page — batch slug resolution (N+1 fix)
**File:** `apps/web/src/pages/Notify.tsx`
- Replace per-notification sequential fetches with 2 batch requests: `GET /threads?id=in.(...)&select=id,board_id` + `GET /boards?id=in.(...)&select=id,slug`
- Reduces worst case from 40 requests to 2
